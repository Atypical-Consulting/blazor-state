# BlazorStatePlus Source Generator Design

## Goal

Replace manual plumbing (base class inheritance, string keys, `UseSlice`/`UseGroup` calls) with a `[Slice]` attribute and an incremental source generator that emits all wiring code automatically. One NuGet package, zero ceremony.

## User-Facing API

### Simple case

```csharp
public partial class PersistentCounter : ComponentBase
{
    [Slice]
    private IStateSlice<int> _counter;

    private void Increment() => _counter.Value++;
}
```

### Complex case (async init, dynamic key, TTL)

```csharp
public partial class ProductDetail : ComponentBase
{
    [Inject] private ProductService Products { get; set; } = null!;
    [Inject] private ReviewService Reviews { get; set; } = null!;
    [Parameter] public int ProductId { get; set; }

    [Slice(TimeToLive = "00:05:00")]
    private IStateSlice<ProductPageState> _page;

    [Slice]
    private IStateSlice<int> _viewCount;

    partial void OnInitializeSlices(SliceInitContext ctx)
    {
        ctx.Page
           .KeySuffix(ProductId)
           .InitializeFrom(async () => new ProductPageState
           {
               Product = await Products.GetAsync(ProductId),
               Reviews = await Reviews.GetSummaryAsync(ProductId),
               IsInWishlist = await Products.IsInWishlistAsync(ProductId)
           });

        ctx.ViewCount
           .DefaultValue(0);
    }

    private void ToggleWishlist()
    {
        var state = _page.Value;
        _page.Value = state with { IsInWishlist = !state.IsInWishlist };
    }
}
```

### Key design decisions

- Fields are typed `IStateSlice<T>` — users keep full access to `WasRestored`, `IsDirty`, `IsStale`, `OnChanged`.
- `[Slice]` is the only marker needed. No base class. No `IStateGroup`. No string keys by default.
- Keys are auto-derived: `"{ClassName}.{fieldName}"` (e.g., `_counter` on `PersistentCounter` becomes `"PersistentCounter.counter"`).
- `KeySuffix(params object[] parts)` appends segments separated by `":"` (e.g., `"ProductDetail.page:42"`). `KeyOverride(string)` is the escape hatch for fully custom keys.
- Static config (TTL, AllowUpdatesOnNavigation) lives on the attribute. Dynamic config (keys, factories) lives in `OnInitializeSlices`.
- `OnInitializeSlices` is optional — only implement it if you need async init or dynamic config.
- No `IStateGroup` marker interface — complex types work the same as primitives through `IStateSlice<T>`.

## SliceAttribute

```csharp
[AttributeUsage(AttributeTargets.Field)]
public sealed class SliceAttribute : Attribute
{
    public string? TimeToLive { get; set; }
    public bool AllowUpdatesOnNavigation { get; set; }
}
```

## SliceBuilder\<T\>

Runtime config holder used by the generated `SliceInitContext`:

```csharp
public sealed class SliceBuilder<T>
{
    public SliceBuilder<T> KeySuffix(params object[] parts);
    public SliceBuilder<T> KeyOverride(string key);
    public SliceBuilder<T> DefaultValue(T value);
    public SliceBuilder<T> InitializeFrom(Func<Task<T>> factory);

    // Used by generated code — hidden from IntelliSense
    [EditorBrowsable(EditorBrowsableState.Never)] public string ResolveKey(string baseKey);
    [EditorBrowsable(EditorBrowsableState.Never)] public T GetDefaultValue();
    [EditorBrowsable(EditorBrowsableState.Never)] public bool HasAsyncFactory { get; }
    [EditorBrowsable(EditorBrowsableState.Never)] public Func<Task<T>> GetAsyncFactory();
    [EditorBrowsable(EditorBrowsableState.Never)] public Action<StateSliceOptions>? BuildOptions(Action<StateSliceOptions>? attributeDefaults = null);
    [EditorBrowsable(EditorBrowsableState.Never)] public async Task InitializeAsync(IStateSlice<T> slice);
}
```

## Generated Code Shape

For the `ProductDetail` example above, the generator emits:

```csharp
// <auto-generated/>
public partial class ProductDetail : IDisposable
{
    [Inject] private StateManager __sm { get; set; } = null!;

    private SliceInitContext? __ctx;

    protected override void OnInitialized()
    {
        base.OnInitialized();

        var ctx = new SliceInitContext();
        OnInitializeSlices(ctx);

        _page = __sm.CreateSlice<ProductPageState>(
            ctx.Page.ResolveKey("ProductDetail.page"),
            ctx.Page.GetDefaultValue(),
            ctx.Page.BuildOptions(o => o.TimeToLive = TimeSpan.Parse("00:05:00")));

        _viewCount = __sm.CreateSlice<int>(
            ctx.ViewCount.ResolveKey("ProductDetail.viewCount"),
            ctx.ViewCount.GetDefaultValue(),
            ctx.ViewCount.BuildOptions());

        __ctx = ctx.HasAsyncInit ? ctx : null;
    }

    protected override async Task OnInitializedAsync()
    {
        if (__ctx is null) return;

        await __ctx.Page.InitializeAsync(_page);
        await __ctx.ViewCount.InitializeAsync(_viewCount);

        __ctx = null;
    }

    partial void OnInitializeSlices(SliceInitContext ctx);

    public void Dispose()
    {
        _page?.Dispose();
        _viewCount?.Dispose();
    }

    public sealed class SliceInitContext
    {
        public SliceBuilder<ProductPageState> Page { get; } = new();
        public SliceBuilder<int> ViewCount { get; } = new();
        public bool HasAsyncInit => Page.HasAsyncFactory || ViewCount.HasAsyncFactory;
    }
}
```

### Generated code properties

- `OnInitializeSlices` called exactly once — ctx is stashed only if async init is needed, nulled out after.
- No `PersistentComponentState` injection — `StateManager` already owns persistence internally.
- Attribute-level options (TTL) baked as literals in generated code; `BuildOptions()` layers on runtime config from the builder.
- Early return in `OnInitializedAsync` — zero-cost when no async factories exist.

### Conflict resolution

- If the user implements `IDisposable`, the generator emits `__DisposeSlices()` instead and the user calls it from their `Dispose`.
- If the user overrides `OnInitialized`, the generator emits `partial void OnAfterSlicesCreated()` as a hook instead.

## Source Generator Architecture

### Project structure

```
BlazorStatePlus/
  BlazorStatePlus.csproj              (existing library, ships runtime types)
  Attributes/
    SliceAttribute.cs
  Generators/
    SliceBuilder.cs                   (runtime helper)
BlazorStatePlus.Generators/
  BlazorStatePlus.Generators.csproj   (analyzer/generator, packed into main nupkg)
  SliceIncrementalGenerator.cs
  ComponentModel.cs                   (parsed representation of a component)
  Emitter.cs                          (code generation templates)
```

### How it ships

The generator project targets `netstandard2.0` (Roslyn requirement) and is referenced with `OutputItemType="Analyzer"` in the main csproj so it ships inside the same NuGet package. Consumers add one package reference.

### Generator pipeline

`SliceIncrementalGenerator` is an `IIncrementalGenerator`:

1. **Syntax filter** — find field declarations with `[Slice]` attribute.
2. **Semantic transform** — resolve containing class, field types, attribute parameters. Build a `ComponentModel` per class.
3. **Emit** — `Emitter` generates the partial class source for each `ComponentModel`.

## Compile-Time Diagnostics

| ID | Severity | Condition |
|---|---|---|
| `BSP001` | Error | `[Slice]` on a non-partial class |
| `BSP002` | Error | `[Slice]` field type is not `IStateSlice<T>` |
| `BSP003` | Error | Class with `[Slice]` doesn't inherit `ComponentBase` |
| `BSP004` | Warning | `OnInitializeSlices` defined but no `[Slice]` fields found |
| `BSP005` | Error | `[Slice]` attribute has invalid `TimeToLive` format |
| `BSP006` | Warning | Component already implements `IDisposable` — emit `__DisposeSlices()` helper instead |
| `BSP007` | Error | Duplicate resolved keys — two `[Slice]` fields that would produce the same key |
| `BSP008` | Error | `[Slice]` on a static field |
| `BSP009` | Error | `[Slice]` on a property instead of a field |
| `BSP010` | Error | `OnInitializeSlices` has wrong signature |
| `BSP011` | Warning | Component already overrides `OnInitialized` — generator emits `OnAfterSlicesCreated()` hook instead |
| `BSP012` | Warning | `[Slice]` field has an initializer — it will be overwritten by the generator |
| `BSP013` | Error | `T` in `IStateSlice<T>` is not JSON-serializable |

No runtime error handling beyond what `StateManager` already provides. The generator makes invalid states unrepresentable at compile time.

## Changes to Existing Library

### Removed

- `IStateGroup` — no longer needed; complex types work as `IStateSlice<T>`.
- `PersistentComponentBase` — replaced by the generator.
- `UseSlice`, `UseGroup`, `CreateGroup` methods — the generator calls `CreateSlice` directly.

### Kept as-is

- `IStateSlice<T>`, `StateSlice<T>`, `StateSliceOptions` — unchanged.
- `StateManager.CreateSlice`, `CreateAndInit`, `CreateAndInitAsync` — unchanged.
- `ServiceCollectionExtensions.AddBlazorStatePlus()` — unchanged.

### Added

- `SliceAttribute` — marker attribute with optional static config.
- `SliceBuilder<T>` — runtime config holder for generated `SliceInitContext`.
- `BlazorStatePlus.Generators` project — the incremental source generator.

### Updated

- Examples rewritten to use `[Slice]` attribute.
- `GlobalUsings.cs` — export `SliceAttribute`.

## Testing Strategy

### Generator tests (unit)

Use `Microsoft.CodeAnalysis.CSharp.SourceGeneratorTesting` to verify generated output:

- Single slice, no config — correct generated partial class.
- Multiple slices with mixed config — correct `SliceInitContext` shape.
- Async init — `OnInitializedAsync` override runs factories.
- `KeySuffix` — correct key resolution.
- Attribute options (TTL) — baked into generated code correctly.
- Each diagnostic (`BSP001`–`BSP013`) — fires on the right input, doesn't fire on valid input.

### Runtime integration tests

Use `bunit` to test generated components in a real Blazor test host:

- Slice value survives prerender-restore cycle.
- `WasRestored` is `true` after restore.
- `IsDirty` tracks mutations.
- `IsStale` respects TTL.
- Async factory only runs when not restored.
- Dispose cleans up all slices.
- Component with user-defined `Dispose` + `__DisposeSlices()` works.
- Component with user-defined `OnAfterSlicesCreated()` fires at the right time.

### Snapshot tests

Golden-file comparisons of generated output for key scenarios — catches unintended changes to emitted code shape.
