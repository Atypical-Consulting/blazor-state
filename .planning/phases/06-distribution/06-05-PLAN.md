```razor
@page "/counter-server"
@rendermode InteractiveServer

@using Bustand.Sample.Stores

@*
    Counter Page - Interactive Server Rendering

    KEY CONCEPTS:
    - @rendermode InteractiveServer means UI interactions are processed on the SERVER
    - User clicks button -> SignalR sends event to server -> server updates state -> SignalR sends UI update back
    - State lives in server memory, persisted to browser's LocalStorage via middleware
    - Good for: apps that need server resources, database access, secure operations
    - Tradeoff: requires constant connection, slight latency on interactions

    BUSTAND PATTERN:
    - Inherit from ZustandComponent<TStore, TState> for automatic subscriptions
    - Access Store property to call methods
    - Access State property for current state (automatically updates UI on changes)
*@

@inherits ZustandComponent<CounterStore, CounterState>

<PageTitle>Counter (Server)</PageTitle>

<h1>Counter (Server)</h1>

<p class="render-mode-info">
    <strong>Render Mode:</strong> Interactive Server
    <br />
    <small>State changes are processed on the server via SignalR.</small>
</p>

<div class="counter-display">
    <p>Current count: <strong>@State.Count</strong></p>
</div>

<div class="counter-buttons">
    @* Call store methods directly - they trigger Set() which notifies subscribers *@
    <button class="btn btn-primary" @onclick="Store.Increment">
        Increment
    </button>

    <button class="btn btn-secondary" @onclick="Store.Decrement">
        Decrement
    </button>

    <button class="btn btn-outline-secondary" @onclick="Store.Reset">
        Reset
    </button>
</div>

<div class="counter-custom">
    <h4>Increment by custom amount:</h4>
    <input type="number" @bind="customAmount" min="1" />
    <button class="btn btn-info" @onclick="() => Store.IncrementBy(customAmount)">
        Add @customAmount
    </button>
</div>

<div class="explanation">
    <h4>How it works:</h4>
    <ol>
        <li>This page uses <code>@("@")rendermode InteractiveServer</code></li>
        <li>The CounterStore is registered in DI on the server</li>
        <li>When you click a button, SignalR sends the event to the server</li>
        <li>The store's Set() method updates state and notifies subscribers</li>
        <li>The component re-renders and SignalR sends the new HTML diff to the browser</li>
        <li>Persistence middleware saves to browser's LocalStorage via JS interop</li>
    </ol>
    <p><em>Refresh the page - your count persists!</em></p>
</div>

@code {
    private int customAmount = 5;
}
```

```razor
    BUSTAND WORKS THE SAME:
    - Same ZustandComponent inheritance pattern
    - Same Store.Method() calls
    - Same State.Property access
    - Bustand is rendering-mode agnostic!
```

```razor
@page "/counter-auto"
@rendermode InteractiveAuto

@using Bustand.Sample.Stores

@*
    Counter Page - Interactive Auto Rendering

    KEY CONCEPTS:
    - @rendermode InteractiveAuto is the BEST OF BOTH WORLDS
    - First render: Uses Server rendering (fast initial load, no WASM download wait)
    - Background: .NET WASM runtime downloads in parallel
    - Once ready: Seamlessly switches to WASM (faster interactions, works offline)
    - Good for: Most apps! Fast initial load + eventual offline capability
    - Tradeoff: More complex - need to handle both rendering contexts

    STATE PERSISTENCE ACROSS MODE SWITCH:
    Bustand handles the Server -> WASM transition gracefully:
    - Server render: State persisted to browser's LocalStorage
    - WASM takes over: Reads state from same LocalStorage
    - User sees no interruption - state is consistent!

    SAME COMPONENT CODE:
    Notice this component is IDENTICAL to the Server and WASM versions
    (except the @rendermode directive). That's the power of Bustand's
    mode-agnostic design.
*@

@inherits ZustandComponent<CounterStore, CounterState>
```
